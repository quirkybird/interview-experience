# 山海传媒 - 面试问答

## 面试复盘摘要

### 岗位匹配
山海传媒是做短剧的公司，今年新成立了 AI 事业部，部门约 12-13 人（前后端各占一半，1-2 个全栈）。内部做短剧生产平台，分三条线：视频线、短剧制片线、剧本生成线。制片线用图片/视频/动作模型生成短剧切片，最终用 SDXL 2.0（接官方）；剧本线做 IP 改编（小说转短剧剧本）、爆款迭代、原创剧本等。开发模式是 monorepo、前后端不分岗、一个人从需求拆分到前后端一起开发，和我现在用 Claude Code + MCP 的全栈模式高度一致。匹配度高：他们要的就是会用 AI 全栈开发、懂 RAG/Agent 的人，跟我的方向很合。唯一差距是他们重度用图片/视频生成模型，我只做过文本模型。

### 面试问题
- **AI 编程方式**：PRD + 设计稿 → 语雀 MCP 读文档、Sketch MCP 读设计 → Claude Code plan mode + openSPEC → review 重点流程 → 人工 review 代码 → PR → 测试上线
- **项目形态**：小程序、H5、C 端为主
- **全栈协作方式**：NestJS 后端分模块，先定义前后端交互字段（卡片、流式文本的 stream 类型），拿到 PRD 先整体过一遍设计
- **多模型调度**：左下角选模型（DeepSeek / 通义千问），OpenAI 兼容 或 Anthropic 兼容两种调用方式
- **RAG 全流程**：DocLing 转 Markdown → HeaderSplitter 按标题分级 → 超 500 阈值 + 50 overlap 递归分块 → 批量 embedding → 存 Milvus + metadata（文档名/版本，供 filter）→ 召回时关键词 + 语义双路 → RRF 归一化排序 → rerank 模型（阈值 0.75-0.8）→ 塞进 prompt 模板
- **文件解析噪声**：后端解析，图片用 OCR 模型识别文本再插回
- **Agent 架构中我的部分**：RAG 工具封装、流式输出优化、记忆设计
- **上下文/记忆设计**：三层 —— 当前会话动态窗口（最近 10 条）、超限异步压缩成 summary、长期记忆（个人健康档案，经 MCP 获取）
- **流式渲染优化**：增量更新（已渲染段不重复渲染）、未闭合标签补全、requestAnimationFrame 按帧（16ms）刷内存缓冲
- **流式表格/代码块渲染**：用 remark 类库自动补全未闭合标签再渲染，避免输出中途难看/突变
- **反问**：岗位做什么项目、团队规模

### 表现好坏
✅ **答得顺的**：这场是我目前发挥最好的一场。RAG 全流程讲得非常完整——从 DocLing 转 Markdown、HeaderSplitter 分级、500+50 overlap 递归分块、Milvus + metadata、双路召回、RRF、rerank 阈值，一条龙讲清楚了，面试官明确说"讲得很详细""确实真正做过"。记忆三层架构（动态窗口 / 异步压缩 summary / 长期档案）和流式渲染优化（增量更新 + RAF 按帧 + 未闭合标签补全）也答得有条理。流式表格渲染这个细节问题答上来了，说明实现细节是真懂的。
❌ **卡住/可改进的**：开头有一次没听清面试官问的"基于 LangChain 做的多模型能力"，反问了一句"群里面什么能力"，是 ASR 把 LangChain 听成"链"导致的沟通小卡顿，但很快接上了。多模型只做过文本模型，被追问时如实承认——这在短剧公司是短板，他们重度用图片/视频模型。

### 面试官关注点
面试官非常务实，全程在追技术实现细节（RAG 怎么分块、文件噪声怎么清洗、流式表格怎么渲染、记忆怎么存），目的是验证项目的真实性和深度。他反复说"明白明白""确实真正做过"，说明对真实做过项目的人很认可。他特别在意全栈/不分岗的开发模式能不能接得上，因为他们部门就是 monorepo 一人通吃。对 AI coding 工作流很有共鸣，主动说"我们的开发模式都是非常像的"。

**面试官：** 同学你好，我是这边的面试官，先做个自我介绍吧。

**回答：** 面试官你好，我叫杨青宇，毕业于渤海大学软件工程专业，之前在数字码力担任前端开发，主要负责支付宝健康事业部相关的业务开发。我的工作可以分为两部分：一部分是传统业务开发，包括支付宝医疗健康小程序、健康岛以及支付宝运动的相关业务场景，具体有健康能量的体系建设、权益兑换链路（权益中心）的开发、校园增长活动的搭建以及其他核心功能开发，同时也负责一些通用组件和 SDK 的抽象沉淀。第二部分是参与到一个 AI 智能体相关开发，也是我目前投入比较多的地方，我开发了运动健康的 AI 智能体，从前端到后端都有实践经验。技术栈主要包括 NestJS、MongoDB、Milvus 这些。我主要负责搭建基于 RAG 的运动健康知识库，对运动相关的常见问题、运动规则、权益信息进行向量化处理，并基于 Milvus 实现检索召回。同时封装了一个多模型调度层，支持不同模型调用，比如 DeepSeek、通义千问，此外还实现了一些 MCP server 的调用能力，让智能体能够访问用户的个人信息。并基于 NestJS 实现 SSE 流式输出、卡片和文本的流式渲染、多轮对话以及上下文记忆能力。除了业务开发，我也比较关注工程化和效率提升，比如性能优化、打包优化，最近也在企业中实践 AI coding。往后我更希望能继续往 AI 应用开发或者偏 AI 的相关项目走。

**面试官：** 讲得很详细。我看到你说有在企业里用 AI coding，你现在是怎么用 AI 开发的？整个 AI 编程是一个什么方式？

**回答：** 之前在公司，我们都是产品出 PRD、设计出设计稿。我当时用语雀，语雀有 MCP 可以直接读取文档内容。我们使用的工具是 Claude Code，通过 MCP 获取文档的具体信息。设计稿用的是 Sketch，也是通过 MCP 读取。基本上就是用 plan mode 加 openSPEC 的模式，先计划，然后 review 它重点的流程，最后输出。最后还是会人工 review 一遍代码，然后提交 PR，测试没问题就上线。

**面试官：** 我看你这个项目经历基本上都是小程序之类的，是吗？

**回答：** 对，小程序、H5，这个公司基本上都是 C 端的内容比较多。

**面试官：** 你不仅做了前端，还有全栈开发。你们用 AI、像刚说的用 Claude Code 接 MCP 拿文档拆分需求、计划开发之后，是前后端一起做了吗？还是分开做的？

**回答：** 我们后端用的是 NestJS，类似于 Java 的 Spring 框架，分成一个一个模块，也是分开做的。流程跟传统其实差不多，先定义好前后端交互的字段，比如我们的输出有卡片、流式文本，定义好对应的 stream 里面的类型。还有一个细分环节，就是拿到 PRD 之后，对整体的设计先过一遍。

**面试官：** 看起来挺好的。我看到你基于 LangChain 做的一个多模型能力，这个是怎么做的呢？

**回答：** 多模型指的是，比如左下角有一个选项可以选，你是用 DeepSeek 还是千问。本质就是调用对应的 API，如果是 DeepSeek 就调 DeepSeek 的 API，如果是千问，它们如果兼容的话，可能直接用 OpenAI 的兼容方式，或者 Anthropic 兼容，这两种都比较流行。选用之后它就会调用对应的后端服务。

**面试官：** 你们封装的基本上都是文本模型吗？

**回答：** 对，我们基本上都是做文本接入的。

**面试官：** 你基于 RAG 做了向量化检索，在这个项目里你具体做了什么？说一下这个 RAG 是怎么做的。

**回答：** 我参与到整个流程。我们把 RAG 封装成 Agent 的一个工具，工具里有一个 description，包含运动相关场景的问答、运动规则、权益信息。这是通过产品在系统里上传文档，一般是 PDF、Word 比较多，也会有一些 Excel。我们会统一用一个 DocLing 的库做文档分析，最后统一转成 Markdown 格式，因为 Markdown 本身是结构化的，有一级标题、二级标题。然后通过一个 HeaderSplitter 按多级标题来划分不同的分块。如果某一个分块超过阈值（500），可能加 50 的 overlap，再进行一个递归的文本分块 loader，通过句号、换行符这些特殊字符进行划分，保证每个 chunk 不超过 500 的 size。把这些进行批量 embedding。因为文件可能比较大，我们都是通过异步的方式，会告诉用户现在到什么过程了，比如文档解析完成、embedding 完成。分块之后存进 Milvus 向量库，还会带一些 metadata，比如文档的名称、是哪个版本，这些信息可以在后续用 filter 筛选，确定一个更精确的范围，也可以用来生成来源参考。召回的时候，用户输入 query，我们会进行两层匹配，一层关键词匹配，一层语义匹配。如果需要 RAG 工具的时候就会识别到，比如你想知道运动的某部分规则，它就会召回比如 10 个 chunk。向量检索是语义化的，它只是找回向量相似度比较相近的内容，所以还有一个关键词检索。经过这两路，通过 RRF 归一化排序，两边都是 10 条，相加得到一个结果，最后用一个 rerank 模型进行精排，可能最后有一个 0.75 到 0.8 的阈值保留这些分数，最后再把这些 chunk 全部塞进 prompt 模板，发给 LLM 输出。

**面试官：** 你传一些文件去进行解析，里面可能包含页眉页脚这些，是前端解析还是后端解析？

**回答：** 这个是后端解析。先用一个 image 的标识放在原文档里面，再通过 OCR 模型识别它的文本，然后插回去再分块，可能会遇到这些。

**面试官：** 对接智能体对话，这个相当于做了一个 Agent 吗？整个 Agent 架构设计里你做了什么？里面有没有相关的工具应用是你做的？

**回答：** 我肯定只是负责其中一部分，就是刚才说的 RAG 是一块，包括它的流式输出优化，还有记忆的设计。

**面试官：** 上下文存储是吧。这个需要讲一下，上下文是怎么存储的？

**回答：** 其实所有的工程最后也就是一个上下文工程，你要保留关键的信息。因为 LLM 是无状态的，我们会有三层：一层是当前会话内容，是一个动态窗口，基本上取双方对话最近的 10 条塞入对话；如果超过 10 条，或者模型设置的 token 数不够了，我们就会进行一个异步的压缩，在后端生成一个 summary，会设计一个 prompt 说明怎么总结，列出哪些点——比如还没处理完的任务、已经处理完的任务这类。所以一个是摘要，一个是会话。还有一个是长期记忆，比如你这个人的一些信息，或者你之前告诉我的信息，我们从个人健康档案（通过 MCP 获取）拿到信息长期储存在你的个人档案里面，然后把这些信息都塞回给模型，这就是我们做的记忆相关的。然后还有流式优化相关的：因为模型如果每次后端推送每个字符都触发重复渲染，对性能会有影响，而且会轻微漂移或闪烁。我们通过增量更新——因为它是 Markdown，有一些标签转换，之前是用换行符分段，我们通过增量更新让上面已经完成渲染的不重复渲染，下面最新的没有完全闭合的，给它加上闭合的标签再渲染最新一条。还有用 requestAnimationFrame，根据帧（比如 16ms）把之前来的字符全部存到前端的内存缓冲里面，每一帧去渲染，就不会后端来一个字就渲染一个。

**面试官：** 你提到 SSE 流式返回的时候，如果返回的是表格，怎么渲染？表格在中间可能还没闭合，输出期间你是怎么展示的？

**回答：** 这个就是我刚才说的，比如是一个代码块或者表格，它识别到有一定的特征——代码块是三个反引号加一个语言名称，我们后面有一个库叫 remark，它会自动补充掉那一条未闭合的，再让 Markdown 解析库去渲染。相当于自动识别，把后面那个标签补充上，就不会说这一条很难看、最后表格输出完成时突然一下变成那种样子。

**面试官：** 所以是用库去判断、怎么匹配这个格式的吧？

**回答：** 是用那个库，它的原理就是动态补充闭合标签。

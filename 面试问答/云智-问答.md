**面试官：** OK 啊，那我们就从 React 基础架构这块开始聊。React 的话，避不开的就是 hooks 嘛。然后可以先简单讲一讲 hooks 的原理，然后以及你在用 hooks 的时候，有没有一些踩坑的，或者觉得不合理的地方？

**回答：** React hooks 是 React 16.8 之后函数式组件引入的 hooks。它的原理其实就是按每个组件来划分。每个组件我们一般是在顶层使用那个 hooks 嘛，不会在一些嵌套逻辑、判断逻辑、循环逻辑里面使用。它的主要原理是因为针对于每个组件，也就是相当于说每个工作单元就是每个 Fiber 来说，它会把这个 hooks 的声明加载到一个链表。React 只知道每个 hooks 是哪个（比如 useState 或 useEffect），以及它的顺序，然后它不会记录说你那个值到底是什么样。所以它是根据类似于索引的机制来更新 hooks 的值。如果你这个组件发生了重复渲染之后，它拿到原来的值，是根据链表索引拿到的，不是根据 hooks 本身的名称。然后使用的一些坑的话，就比如刚才说到的不能在循环这些里面使用，因为如果在那里面使用的话，索引本身对应的 hooks 可能就对不上，就会引发一些意外的渲染错误。然后还有闭包陷阱，就是 useEffect 的依赖数组。假如你的依赖数组是空数组，这个值本身应该是 state，应该是响应式的，但是在你声明的那一刻，它那个对应函数或逻辑的值就在这一次锁死了。下次你可能想更新这个值，但没有传入依赖数组，这个值就拿不到最新的值。这个时候一般是用 useRef 来解决。

**面试官：** 嗯，OK 挺详细的。看来你们平时用 React 其实挺多的。那我想想你平时会把什么样的东西封装为 hooks，或者你们业务中 hooks 有没有什么规范？比如说什么样的东西，我要把它封装成 hooks，设计上要注意什么？

**回答：** 这个具体的我们这边没有看到有什么规范。但是我个人理解很容易混淆的就是和工具函数 utils 混。我这边更倾向于你需要依赖其他基础 hooks 这种能力的场景，你才把它归为 hooks。就比如你要根据某个值的更新，然后更新它内部的一些东西。比如说之前封装一个订阅消息的 hooks，你肯定有一个类似于 ID 的东西，可能会提供一些方法去订阅、取消订阅。订阅之后你肯定想拿到一个订阅后的状态，就是用户到底选择订阅还是没订阅。这个值就需要动态的去更新，这时候可能就会用到 useEffect。所以 hooks 需要依赖基础的一些 hooks 的能力，这种我们就会把它封装成 hooks。如果是一些更通用的能力，可能就单独封装成一个 hooks 库。

**面试官：** 就是它和 utils 有区别吗？就比如说有些情况下，什么情况下它只能用 hooks，不能用 utils。除了就像你说的要用到 hooks 之外，还有哪些情况下必须用 hooks，而不是用 utils？

**回答：** 必须用 hooks？我想一下，这个我暂时想不到。

**面试官：** 嗯嗯，就是其实我想说的可能就是一些你要用一些状态变量相关的嘛，你要存储一些状态。比如说你需要像常规的 hooks 里面有一些数据状态的扭转，可能在多个模块之间。

**回答：** 对，反正其实我刚才也有提到，比如一个订阅消息，你肯定要知道这个订阅消息的一个实时状态，就是那个值是响应式的，订阅了之后就不展示这个按钮了这些。

**面试官：** 嗯，是是像这种情况下，就是只能用 hooks，不能用 utils 的。对对对。
